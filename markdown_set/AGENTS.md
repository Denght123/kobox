# AGENTS.md

## 目的

本文件用于约束 Codex 在本仓库中的工作方式，确保它输出的代码：

- 与项目的产品定位一致
- 与现有技术栈一致
- 易读、易维护、易扩展
- 不擅自引入无关复杂度
- 不偏离当前开发阶段的真实需求

本项目是一个**动漫收藏管理与展示网站**，不是社区、不是论坛、不是评分站、不是逐集追番系统。

---

## 1. 项目定位

请始终围绕以下核心目标工作：

1. 用户可以注册、登录并维护个人资料
2. 用户可以搜索动漫并加入自己的收藏
3. 用户可以管理收藏状态：
   - 已看完
   - 在追
   - 想看
   - 搁置
   - 弃番
4. 用户可以维护“最喜欢的动漫排名”
5. 用户拥有一个可公开访问的收藏柜主页
6. 网站强调：
   - 二次元审美
   - 收藏展示感
   - 封面墙
   - 个人空间
   - 分享能力

不要把项目实现成：

- 社区论坛
- 评论系统
- 私信系统
- 关注流
- 动态 feed
- 逐集进度记录系统
- 年度/季度报告系统
- 复杂推荐系统

如果需求没有明确提出，请不要主动扩展为这些方向。

## 4. 前端代码规范

### 4.1 页面与组件设计
页面应按实际业务拆分，不要把所有逻辑堆到一个页面里。

推荐页面级目录：
- `pages/AuthPage.vue`
- `pages/HomePage.vue`
- `pages/SearchPage.vue`
- `pages/FavoriteRankPage.vue`
- `pages/SettingsPage.vue`
- `pages/PublicProfilePage.vue`

推荐组件级目录：
- `components/anime/AnimeCoverCard.vue`
- `components/collection/CollectionSection.vue`
- `components/profile/ProfileHeader.vue`
- `components/share/ShareQRCodeCard.vue`

### 4.2 组件职责
每个组件应尽量职责单一：

- 展示组件只做展示
- 业务逻辑放到 composables / stores / query hooks
- 不要在深层子组件里写复杂请求流程
- 不要把多个页面的逻辑耦合在一起

### 4.3 组合式 API
优先使用 Vue 3 Composition API 和 `<script setup lang="ts">`。

避免：
- 旧式 Options API 写法
- 混乱的响应式变量命名
- 过多 `watch` 叠加

### 4.4 状态管理
状态管理遵循：

- 本地 UI 状态：组件内处理
- 页面级共享状态：Pinia
- 服务端数据状态：TanStack Query
- 不要用 Pinia 替代所有接口缓存
- 不要把整个页面所有数据都塞进一个巨大的 store

### 4.5 请求层
请求相关规范：

- 所有接口请求放在 `src/api/`
- 统一 Axios 实例
- 统一错误处理
- 请求函数要有明确返回类型
- 不要在页面中直接硬写裸请求逻辑

### 4.6 国际化
所有 UI 文案必须可国际化：

- 不要把用户可见文案直接写死在组件中
- 使用 `vue-i18n`
- 默认文案优先简体中文
- key 命名清晰，如：
  - `auth.login`
  - `auth.register`
  - `home.collectionCount`

### 4.7 样式规范
样式应符合产品定位：

- 清新、二次元、展示感强
- 不要做成后台管理系统风格
- 不要使用与产品调性冲突的暗黑极客风默认样式
- 首页要突出封面墙和收藏柜视觉

如果进行样式开发：
- 优先保持页面层次清晰
- 避免过度嵌套 CSS
- 尽量保证组件样式局部可控

## 5. 后端代码规范

### 5.1 分层结构
后端必须保持清晰分层，推荐结构：

- `api/`：路由层
- `schemas/`：Pydantic 输入输出模型
- `models/`：SQLAlchemy ORM 模型
- `services/`：业务逻辑
- `repositories/`：数据访问
- `core/`：配置、鉴权、基础能力
- `db/`：数据库连接与 session

不要把：
- 路由
- SQL 查询
- 业务逻辑
- 数据校验
全部写在一个文件里。

### 5.2 路由层职责
路由层只负责：

- 参数接收
- 调用 service
- 返回 response schema
- 处理基础鉴权依赖

不要在路由函数里直接堆复杂业务逻辑。

### 5.3 Service 层职责
Service 层负责：

- 业务规则
- 多表协作
- 状态流转
- 权限校验
- 聚合返回结果

### 5.4 Repository 层职责
Repository 层负责：

- ORM 查询
- 查询条件封装
- 基础增删改查

不要在 repository 层混入前端视图逻辑。

### 5.5 Schema 规范
所有请求体和响应体都必须定义 Pydantic schema：

- 创建 schema
- 更新 schema
- 返回 schema
- 列表 schema
- 公开页 schema

不要直接把 ORM 模型返回给前端。

### 5.6 数据库迁移
如果修改 ORM 模型，必须同步考虑 Alembic migration。

不要：
- 只改 model 不改 migration
- 手写临时 SQL 而不更新迁移
- 让数据库结构和代码脱节

## 6. 命名规范

### 6.1 前端命名
- 组件名使用 PascalCase
- composable 使用 `useXxx`
- store 使用 `useXxxStore`
- 类型定义使用明确语义命名
- 避免 `data`, `item`, `list` 这类模糊命名泛滥

### 6.2 后端命名
- 模型类使用 PascalCase
- 表名使用清晰复数或项目统一约定
- schema 名称要体现用途，如：
  - `UserProfileUpdate`
  - `AnimeSearchResult`
  - `PublicCollectionItem`

### 6.3 枚举值
后端枚举优先使用英文稳定值，前端做本地化展示。

例如收藏状态：

- `completed`
- `watching`
- `plan_to_watch`
- `on_hold`
- `dropped`

不要在接口协议里直接返回中文枚举值。

---

## 7. 数据与接口约束

### 7.1 时间格式
统一使用 ISO 8601。

### 7.2 列表返回
统一包含：

- `items`
- `total`
- `page`
- `page_size`

如果当前接口不需要分页，也要保持风格一致，不要随意变化。

### 7.3 错误返回
错误结构尽量统一，至少包含：

- `code`
- `message`
- `details`（可选）

### 7.4 公开接口与私有接口分离
公开主页相关接口必须与用户编辑接口分离。

例如：
- `/api/public/users/{username}`
- `/api/public/users/{username}/collections`

不要把“我的主页编辑数据接口”直接暴露给公开访问场景复用。

## 10. 输出代码时的要求

### 10.1 不要输出伪代码
除非用户明确只要思路，否则默认应提供可落地代码。

### 10.2 不要省略关键部分
不要用：
- “此处省略”
- “你可以自行实现”
- “略”

关键逻辑要写完整。

### 10.3 注释要求
只在必要时写注释，避免废话式注释。

好的注释应该解释：
- 为什么这么做
- 特殊边界情况
- 业务规则原因

不要写显而易见的逐行翻译式注释。

## 12. 视觉与产品一致性要求

这是一个二次元收藏展示网站，不是企业管理后台。

所以在涉及前端实现时，Codex 需要注意：

- 首页是“收藏柜”，不是“数据面板”
- 核心视觉是“封面墙”
- 用户的个人主页是核心资产
- 页面应服务于“展示”和“分享”
- 即使写的是功能代码，也不要把结构做得像冷冰冰的 admin dashboard
